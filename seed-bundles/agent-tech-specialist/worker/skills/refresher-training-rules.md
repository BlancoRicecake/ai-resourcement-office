# Refresher Training Rules — 보수 교육 제안 규칙

신기술이 팀 번들(직무 분석가/에이전트 메이커/에이전트 훈련소)의 규칙·스킬을 낡게 만들면, 업데이트 후보를 `보수 교육 제안서`로 작성하는 규칙이다.

훈련소와의 구분: 훈련소는 신임 에이전트에게 직무 지식을 가르치고, 보수 교육은 **팀 번들 자체의 제작 규칙**을 갱신한다.

## 발동 조건

다음 중 하나에 해당할 때만 제안한다.

1. 팀 번들의 규칙이 전제하는 기술·사양이 공식적으로 폐기·대체되었다.
2. 규칙이 다루는 문제에 더 나은 표준 방식이 **안정 단계**에 도달했다.
3. 규칙대로 하면 실제 문제(오류, 보안, 비용 낭비)가 생기는 것이 확인되었다.
4. 규칙에 없는 새 영역(새 도구 유형, 새 권한 모델)이 **조기 채택 이상**으로 자리 잡아 규칙 공백이 생겼다.

발동 조건 미달이면 제안하지 않고 TECH-RADAR.md 추적으로만 남긴다. 기준: **성숙도 조기 채택 이상 + 신호 강도 중 이상**. 실험적 기술로 팀 규칙을 흔들지 않는다.

## 제안 전 필수 절차

1. 대상 번들의 현재 규칙 파일(skills/memory)을 **실제로 읽는다.** 읽지 않고 추측으로 제안하지 않는다.
2. 낡았다고 판단한 부분을 원문 그대로 인용한다.
3. 변경 근거가 되는 기술 변화의 출처·신호 강도·성숙도를 확인한다.

## 제안서 형식

```txt
# 보수 교육 제안서 — [대상 번들]

1. 대상 번들 / 대상 파일
   - 번들: (예: agent-maker)
   - 파일: (예: worker/skills/tool-interview-rules.md)

2. 현재 규칙 (해당 부분 인용)

3. 변경 근거
   - 어떤 기술 변화가 왜 이 규칙을 낡게 만들었나
   - 신호 강도: 강/중 · 성숙도: 조기 채택/안정 · 출처

4. 업데이트 후보 diff
   - 현재: (기존 문구)
   + 변경안: (새 문구)

5. 리스크와 롤백
   - 변경 시 리스크, 문제가 생기면 되돌리는 방법

6. 사용자 컨펌 지점
   - 반영 여부, 반영 담당(사용자/해당 번들 담당), 버전 올림 여부
```

## 자동 수정 금지

- **다른 번들의 파일을 직접 수정하지 않는다.** 산출물은 항상 제안서다.
- 반영은 사용자가 승인한 뒤 사용자(또는 해당 번들 담당 직원)가 수행한다.
- 반영이 확정되면 대상 번들의 `bundle.json` 버전 올림(규칙 변경은 minor)을 함께 안내한다.
- 제안·승인·거절 이력은 `memory/DECISIONS.md` 후보로 남긴다. 거절된 제안은 새 변화 없이는 다시 내지 않는다.

## 제안 빈도 제어

- 같은 번들에 대한 제안은 모아서 낸다. 뉴스 하나마다 제안서를 만들지 않는다.
- 사소한 문구 개선은 보수 교육 대상이 아니다. 규칙의 전제가 바뀐 경우만 다룬다.
