# Tech Radar Rules — 기술 추적 규칙

기술 전문가는 아무 뉴스나 브리핑하지 않는다. 팀 업무에 영향을 주는 신호만 골라, 신호 강도와 성숙도를 붙여 추적한다.

## 추적 범주 (5개)

| 범주 | 추적 대상 예 |
| --- | --- |
| MCP 생태계 | 새 MCP 서버, MCP 사양 변경, 인증·전송 방식 변화 |
| 런타임 신기능 | Claude Code·Codex 등의 새 기능, 설정 방식, 훅·스킬·서브에이전트 체계 변화 |
| 모델 업데이트 | 새 모델 출시, 가격·컨텍스트 변경, 기존 모델 폐기 예고 |
| 에이전트 설계 패턴 | 멀티 에이전트 구성, 메모리 설계, 컨텍스트 관리, 평가 방법론 |
| 도구 연결·권한 기술 | 권한 모델, 샌드박스, 시크릿 관리, 파일·API 접근 제어 |

5개 범주 밖의 소식은 원칙적으로 추적하지 않는다. 범주 신설이 필요하면 사용자에게 제안한다.

## 신호 강도 판정 (강/중/약)

- **강**: 공식 문서·릴리스 노트·개발사 공식 발표로 당일 확인됨.
- **중**: 신뢰할 만한 복수의 독립 소스에서 일관되게 언급되나 공식 확인은 아직 없음.
- **약**: 단일 소스, 커뮤니티 소문, 예고·루머 수준.

규칙: 수치를 직접 확인할 수 없으면 숫자를 지어내지 않고 신호 강도로만 표현한다. 약 신호는 브리핑의 "계속 추적 중"에만 두고, 권장 액션의 근거로 쓰지 않는다.

## 성숙도 단계

- **실험적**: 나온 지 얼마 안 됐고 사양이 바뀔 수 있음. 개인 실험 대상.
- **조기 채택**: 실사용 사례가 나오고 있으나 표준은 아님. 파일럿 적용 후보.
- **안정**: 공식 지원·문서화가 갖춰지고 널리 쓰임. 팀 표준 규칙 반영 후보.

성숙도는 보수 교육 제안의 발동 조건과 연결된다: 조기 채택 이상만 제안 근거가 된다.

## 팀 적용 가능성 평가

각 동향 항목에 대해 `memory/PROJECT.md`의 팀 맥락 기준으로 판단한다.

1. 팀의 런타임·번들 형태에 해당되는 변화인가?
2. 팀의 3대 파이프라인 단계(설계/제작/훈련) 중 어디에 영향을 주는가?
3. 지금 액션이 필요한가, 추적만 하면 되는가?

판정 표기: `즉시 검토 / 파일럿 후보 / 추적만 / 무관(버림)`.

## 소음 걸러내기

다음은 브리핑에 넣지 않는다 (필요 시 "버린 소음"에 한 줄로만 남긴다).

- 팀의 런타임·번들 형태와 무관한 소식
- 마케팅성 발표 (구체적 기능·사양 변화가 없는 것)
- 신호 강도 약인데 액션을 요구하는 것처럼 포장된 소문
- 이미 DECISIONS.md에서 거절된 기술의 반복 소식 (새 변화가 없으면)

"이번 주는 유의미한 변화 없음"도 정당한 브리핑이다. 억지로 항목을 채우지 않는다.

## TECH-RADAR.md 연동

- 브리핑 전에 `memory/TECH-RADAR.md`를 읽고, 추적 중 기술의 신호·성숙도 변화를 우선 확인한다.
- 새로 추적할 가치가 있는 기술은 TECH-RADAR.md 기록 후보로 제안한다 (자동 확정 금지).
- 신호가 사라졌거나 폐기된 기술은 판정을 "종료"로 바꾸는 후보를 제안한다.
