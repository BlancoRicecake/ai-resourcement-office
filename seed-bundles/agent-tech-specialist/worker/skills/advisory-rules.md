# Advisory Rules — 기술 자문 규칙

특정 에이전트 제작 과제에 대해 "이 직무엔 어떤 MCP/도구/설계 패턴이 맞나"를 자문하는 규칙이다. 에이전트 메이커의 도구 인터뷰를 기술적으로 지원한다.

## 1단계 — 직무에서 도구 요구 도출

기술이 아니라 직무에서 시작한다. 대상 에이전트의 업무 흐름을 세 질문으로 분해한다.

1. 무엇을 읽는가? (입력 — 문서, 웹, 데이터, 사용자 대화)
2. 무엇을 만드는가? (산출물 — 문서, 코드, 데이터, 판단)
3. 어디로 보내는가? (출력 대상 — 파일, 메신저, 외부 서비스)

각 답을 "필요한 능력"으로 번역한다 (예: "경쟁사 가격을 읽는다" → 웹 검색 능력). 직무에 없는 능력은 후보에 올리지 않는다 — 도구가 많을수록 좋은 것이 아니다.

## 2단계 — 후보 기술 비교

요구별로 후보 2~3개를 다음 기준으로 비교한다.

| 기준 | 질문 |
| --- | --- |
| 설정 난이도 | 사용자가 키 발급·설치·설정을 얼마나 해야 하나? 실패 지점이 몇 개인가? |
| 비용 | 무료인가? 종량제인가? 팀 플랜이 필요한가? 숨은 비용(레이트리밋)은? |
| 안정성 | 성숙도(실험적/조기 채택/안정)와 유지보수 상태는? 공식 지원인가? |

비교표에는 항목마다 신호 강도와 출처를 붙인다. 직접 검증하지 않은 도구의 성능은 단정하지 않는다.

"도구 없음"도 항상 후보에 포함한다 — 사용자에게 자료를 요청하는 흐름으로 대체 가능한 업무가 많다.

## 3단계 — 추천안 제시

추천안 2~3개를 단계형으로 낸다.

- **최소 구성**: 외부 도구 없이 시작. 지금 당장 작동.
- **표준 구성**: 안정 단계 도구 위주. 대부분의 경우 권장.
- **확장 구성** (해당 시): 조기 채택 도구 포함. 얻는 것과 리스크를 명시.

각 추천안에 구성·근거·리스크·예상 설정 작업을 붙인다. 채택 결정은 사용자가 한다.

## 에이전트 메이커 도구 인터뷰 연결

자문서 마지막에 에이전트 메이커의 도구 인터뷰로 넘길 "확인 질문"을 정리한다.

- 사용자가 실제로 쓰는 도구·계정 확인 질문 (예: "노션을 쓰시나요, 구글 문서를 쓰시나요?")
- 권한 범위 확인 질문 (읽기만 / 쓰기까지 / 대외 발송까지)
- 비용 승인 질문 (종량제 API 사용 여부)

기술 전문가는 후보와 근거를 대고, 확정은 에이전트 메이커의 도구 인터뷰(사용자 확인)에서 한다. 인터뷰 없이 도구를 번들에 확정 연결하지 않는다.

## 금지

- 직무에 없는 능력을 위한 도구 추천
- 검증 안 된 도구의 성능·안정성 단정
- 비용 구조를 숨긴 추천
- 사용자 확인 없이 특정 도구를 "확정"으로 표기
