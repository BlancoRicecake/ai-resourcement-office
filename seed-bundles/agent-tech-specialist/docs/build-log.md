# Build Log

## Source Problem

에이전트 제작 팀(직무 분석가 → 에이전트 메이커 → 에이전트 훈련소)에 최신 기술이 들어오는 채널이 없었다. MCP 생태계, 런타임 신기능, 모델 업데이트, 설계 패턴, 도구 연결·권한 기술은 계속 바뀌는데 아무도 전담 추적하지 않아, 팀 번들의 규칙·스킬이 조용히 낡고 제작 과제의 도구 선택이 각자의 기억에 의존했다.

## MVP Scope

팀 전체를 지원하는 기술 참모를 만든다. 3대 산출물: (1) 팀에 유의미한 것만 골라낸 기술 동향 브리핑, (2) 제작 과제별 MCP/도구/설계 패턴 자문서, (3) 낡아진 팀 번들 규칙의 보수 교육 제안서(diff 형태, 자동 수정 금지).

## Design Inputs From This Session (2026-07-09)

- 파이프라인의 특정 단계가 아니라 팀 전체를 지원하는 **참모** 포지션으로 설계 — 분석가에게는 설계 고려사항, 메이커에게는 도구 자문, 훈련소에게는 방법론 변화를 공급한다.
- 훈련소와의 역할 구분을 명문화: 훈련소는 신임 에이전트의 직무 지식, 기술 전문가는 팀 자체의 제작 기술.
- SaaS 리서치 에이전트의 리서치 규칙을 이식: 당일 웹 검색 확인, 신호 강도(강/중/약) 정직 표기, 숫자 지어내지 않기, 추측/사실 구분, 검색 불가 시 "재확인 필요" 표시.
- 소음 필터를 핵심 가치로: 뉴스 요약기가 아니라 팀 업무 영향 기준으로 걸러낸다. "유의미한 변화 없음"도 정당한 브리핑.
- 보수 교육은 성숙도 조기 채택 이상 + 신호 강도 중 이상일 때만 발동, 다른 번들 파일 직접 수정 금지(항상 제안서).
- 직무 특화 누적 메모리 `TECH-RADAR.md` 도입 — 세션을 넘어 기술의 신호·성숙도 변화를 이어서 추적한다.
