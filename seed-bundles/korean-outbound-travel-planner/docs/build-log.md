# Build Log

## Source problem

한국 출발 여행자는 항공, 숙소, 리뷰, 장소와 이동 경로를 여러 서비스에서 따로 비교해야 하며 표시 가격의 수하물·세금·취소 조건이 서로 다르다.

## MVP

- 일본·미국 대표 도시
- FastAPI 로컬 웹 앱
- Amadeus 및 Google 어댑터
- Booking.com 검색·외부 예약 및 Skyscanner 항공·숙소 파트너 어댑터
- 영역별 샘플 폴백
- 균형형·비용 절약형·여유형 일정
- 외부 예약 링크와 공식 확인 체크리스트

## Design decisions

- LLM은 설명만 담당하고 숫자·순위·동선은 코드가 계산한다.
- 한 공급자 장애가 전체 계획을 막지 않도록 부분 결과를 허용한다.
- 제휴 승인이 필요한 공급자는 키 필드를 제공하되 작동한다고 가장하지 않는다.
- 사용자 동의가 있을 때만 로컬 SQLite에 취향을 저장한다.

## LA quality pass

- 숙소 유형을 약한 가점이 아니라 강제 필터로 적용한다.
- LA 장소 후보에 권역, 관심사 태그, 우천 적합도와 권장 체류시간을 부여한다.
- 한 날짜는 원칙적으로 한 권역 안에서 구성하고 권역 내 이동시간으로 순서를 조정한다.
- 도착일은 짧게, 출국일은 공항 버퍼로 비우고 나머지 체류일은 후보가 있는 한 모두 채운다.
- 균형형 3~4곳, 비용 절약형 3곳, 여유형 2곳으로 서로 다른 장소 구성과 밀도를 사용한다.
- 폭우 재계획은 야외 후보를 제거한 뒤 실내 후보로 남은 날짜를 다시 채운다.

## Trust and research fallback pass

- 결과를 `live`, `mixed`, `guided_research`, `sample` 모드로 구분한다.
- 여행 API가 없으면 계획용 추정치를 D 등급으로 표시하고 가격·재고를 실시간으로 표현하지 않는다.
- 항공·숙소 비교 플랫폼과 도시 공식 관광·교통 사이트의 조건 기반 검증 링크를 제공한다.
- 필수 방문지를 강제 제약으로 배치하고 우천 재계획에서도 보존한다.
- 웹 검색 도구가 있는 호스트 에이전트만 공식 출처를 조회하며 로컬 앱은 검색 결과 페이지를 스크래핑하지 않는다.
