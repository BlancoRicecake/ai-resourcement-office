# 예시 입력 ① — 기획 단계 서비스 브리프

(시나리오 A: 기획 자문. 화면이 아직 없고 설계 방향·레퍼런스를 요청하는 경우.)

## 서비스 개요

- **서비스명(가칭)**: 모아가계부 (MoA)
- **한 문장 설명**: 여러 은행·카드 내역을 자동으로 모아 한 화면에서 보여 주는 개인 자산 관리 웹 서비스.
- **핵심 가치**: "흩어진 돈을 한눈에" — 계좌·카드·현금을 통합해 이번 달 지출과 남은 예산을 즉시 보여 준다.

## 타깃 사용자

- 20~30대 사회초년생, 여러 은행·카드를 쓰지만 앱을 여러 개 켜기 귀찮아하는 사람.
- 엑셀 가계부는 부담스럽고, 기존 자산관리 앱은 "복잡하고 광고가 많다"고 느끼는 층.
- 주 사용 상황: 출퇴근 지하철에서 휴대폰으로 3분 확인. 데스크톱은 월 1회 정산 때만.

## 핵심 목표(1차 전환)

- 랜딩 방문자가 **회원가입 + 첫 계좌 연결**까지 도달하는 것.
- 재방문자는 "이번 달 지출·남은 예산"을 3초 안에 확인하고 이탈하는 것(빠른 확인이 핵심 가치).

## 플랫폼

- 모바일웹 우선(주 사용), 데스크톱 웹 보조(월 정산).
- 네이티브 앱은 2차 로드맵. 지금은 반응형 웹으로 시작.

## 알려진 제약

- 초기 팀에 전담 디자이너가 없어, 공개 디자인 시스템/컴포넌트를 최대한 활용하려 한다.
- 금융 데이터라 신뢰감이 중요하고, 계좌 연결 단계의 이탈이 가장 걱정된다.

## 요청

- 이 서비스의 **랜딩페이지와 첫 온보딩(계좌 연결) 설계 방향**을 잡아 달라.
- 참고할 만한 레퍼런스와 활용할 디자인 시스템을 근거와 함께 추천해 달라.
- 화면 제작은 우리가(또는 코딩 에이전트가) 할 테니, 방향과 체크포인트만 자문해 달라.
