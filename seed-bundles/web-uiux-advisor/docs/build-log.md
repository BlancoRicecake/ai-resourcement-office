# Build Log

## Source Problem

웹 화면·기획을 만드는 사람은 많지만, "이 화면이 사용자 목적에 맞는가"를 **현업 프레임워크로 근거 있게 진단**해 주는 채널은 팀에 없었다. 감상평("좀 촌스럽다")과 진단("폴드 아래 밀린 CTA — Fitts·폴드 우선 위반, 심각도 3")은 다르다. 후자는 휴리스틱·법칙·접근성 기준을 인용하고 위치·심각도·개선안으로 환원해야 한다.

## MVP Scope

화면을 **구현하지 않는** 리뷰·자문 전문 직원을 만든다. 두 시나리오: (A) 기획 단계 자문(설계 방향·레퍼런스 추천), (B) 기존 화면 리뷰(현업 체크포인트 진단). 산출물은 Executive Summary → Finding(위치·근거·심각도 0~4) → 우선순위(심각도×수정난이도) → 개선안(Before/After + 텍스트 와이어프레임) → 레퍼런스 근거의 리뷰 리포트. 코드·시각 시안은 범위 밖 — 필요하면 패턴/스킬 이름을 지목해 넘긴다.

## Design Inputs (2026-07-13)

- **리서치 → 설계**: uiux-research 팀원의 리서치 보고서(Nielsen 10대 휴리스틱, Laws of UX, WCAG 2.2 AA, IA·폼·모바일 체크포인트, 레퍼런스 갤러리·디자인 시스템, 한국 시장 관습)를 지식 소스로 삼았다.
- **엔진 없는 지식 그래프**: bundle-standard §2.5 컨벤션에 따라 지식을 `[[위키링크]]` 노드 + INDEX로 적재했다. 코드 엔진 없이 INDEX만 읽고 필요한 노드만 여는 방식. 런타임 0.
- **시드 적재**: 리서치 보고서의 26개 시드 노드 후보를 노드로 적재하고, 보고서가 경고한 dangling 링크 5개([[microcopy-guidelines]], [[above-the-fold-priority]], [[breadcrumbs-pattern]], [[search-vs-browse]], [[aesthetic-usability-effect]])는 **실제 노드로 추가 작성**해 dangling 0을 맞췄다(총 31개 노드, 67 엣지). 링크를 기존 노드로 축소·리다이렉트하는 대신 노드를 추가한 이유: 각 개념이 리뷰에서 독립적으로 인용될 만큼 실전 가치가 있어서다.
- **MengTo/Skills 어댑트(MIT)**: 판단/프로세스형 스킬 3종을 리뷰 관점으로 역변환했다 — `design-first-ui-prompting` → `ui-review-intake`(리뷰 인테이크 질문지), `landing-page` → `landing-page-review`(제작→평가 프레임), `audit-verify-explain-grade-5` → `evidence-based-audit`(리포트 근거 원칙). 구현/시각패턴형은 지식 노드(glass-dark-ui-pattern, visual-style-vocabulary, landing-page-review-checklist)로 이름만 인용하고 구현은 호스트에 넘긴다. 복사가 아니라 패러프레이즈했고, 각 스킬·노드 하단과 README에 출처를 표기했다.
- **역할 경계 명문화**: 리뷰·자문만 하고 구현·실사용자 테스트는 하지 않는다는 경계를 agent.md ④·[[ux-role-boundaries]]에 못박았다. 담당 밖 업무는 협업 연계로 넘긴다 — 카피 전면 작업 → 카피라이팅 분석가(web-copy-analyzer), 코드 구현 → 호스트 코딩 에이전트(+시각 패턴 스킬 지목), 실사용자 테스트 → 설계 제안까지만, 네이티브 앱 idiom 깊이 → 범위 밖(웹 전용, 모바일 웹 포함).
- **성공 기준 S1~S8 명문화(직무 분석가 설계서 기준)**: 졸업 시험 기준의 검증 가능한 성공 기준을 agent.md ③ 말미에 적재했다 — S1 인테이크 없이 진단 시작 금지 / S2 모든 Finding=위치+근거 인용+심각도 0~4(근거 없는 지적 0) / S3 Executive Summary+우선순위(심각도×수정난이도) 포함 / S4 심각도 3+엔 Before/After, 구조 변경엔 텍스트 와이어프레임 ≥1 / S5 구현 코드 0(스킬 지목까지) / S6 검증 못 한 항목 "검증 필요" 표기 / S7 완료 후 메모리 업데이트 후보 제안 / S8 리포트 말미에 복붙 가능한 핸드오프 프롬프트 블록. examples/output 리포트가 이 기준의 답안지 역할을 겸한다.
- **시장 중립 기본값**: 시장·언어 관습은 기본 적용하지 않는다. 인테이크에서 대상 시장이 확인된 경우에만 해당 관습 노드([[korean-market-ux-patterns]])를 적용하도록 agent.md ②·인테이크 스킬·노드를 조건부로 재구성했다.
- **디자인 파일·툴 연동(제한적)**: 웹 전용 포지셔닝은 유지하되, 호스트에 Figma/Pencil MCP·브라우저 조회가 연결되면 디자인 파일·라이브 화면을 직접 읽어 리뷰한다(입력 우선순위 ① 디자인 툴/라이브 URL → ② 스크린샷 → ③ 텍스트 묘사). 네이티브 앱 화면도 파일·스크린샷이 있으면 보편 원칙 수준에서 진단하되 네이티브 고유 관습의 깊이는 제한적임을 리포트에 명시한다.
- **LLM 핸드오프 프롬프트**: 리뷰 결과를 구현자에게 그대로 넘길 수 있는 복붙 프롬프트로 변환하는 스킬(`feedback-handoff-prompt.md`, MengTo `design-first-ui-prompting` 각색)을 추가하고, 워크플로 단계·S8·예시 리포트에 반영했다.

## Pipeline Position

기획/제작 파이프라인을 **평가·자문으로 지원**한다. 기획 단계에는 설계 방향·레퍼런스를, 제작 후에는 화면 진단을 공급한다. 시각 구현이 필요하면 코딩 에이전트/프런트엔드 번들로 넘긴다.
