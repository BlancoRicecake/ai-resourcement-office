# Limitations

- 이 번들은 호스팅 서비스가 아니다. 사용자의 모델 환경에서 직접 실행한다.
- **판단 품질은 호스트 모델에 의존한다** — 결정론 도구가 없으므로 진단·개선안의 질은 이 번들을 구동하는 모델의 능력(추론·비전 해석)에 직접 좌우된다. 약한 모델에서는 Finding의 근거·정확도가 떨어질 수 있다.
- **실사용자 테스트를 대체하지 않는다** — 자문가의 진단은 현업 휴리스틱·법칙에 기반한 **예측**이다. 실제 사용성 테스트·A/B 실험·사용자 인터뷰가 드러내는 문제(멘탈모델 불일치, 실제 전환 영향)를 대신하지 못한다. 중요한 결정은 실사용자 데이터로 검증한다.
- **심미 트렌드는 시효가 있다** — "요즘 미학"(예: 특정 연도의 기술 그리드 트렌드)에 대한 판단은 시간이 지나면 낡는다. 리포트의 심미 관련 항목은 시점 기준임을 명시한다.
- **확인 못 한 것은 확인 못 한 채로** — 정적 스크린샷·텍스트 묘사만으로는 대비비 실측, 로딩 시간, 인터랙션 동작, 링크 클릭 여부를 확정할 수 없다. 이런 항목은 "검증 필요"로 남으며, 정확한 진단은 실제 화면·개발자도구 확인이 필요하다.
- **구현은 범위 밖** — 화면·컴포넌트를 만들지 않는다. 개선안은 텍스트 와이어프레임·섹션 구조·패턴 이름 지목·핸드오프 프롬프트까지다. 실제 제작은 코딩 에이전트/프런트엔드 번들의 몫이다.
- **네이티브 앱은 웹 전용 관점의 제한적 리뷰** — 이 직원은 웹 전용이다(모바일 "웹" 포함). 네이티브 앱 화면은 디자인 파일·스크린샷이 있으면 보편 원칙(위계·대비·타깃 크기·폼·빈 상태) 수준에서 진단할 수 있으나, **네이티브 고유 관습(제스처·플랫폼 내비게이션 idiom)의 깊이는 제한적**이다. HIG/Material 노드는 참조용이며, 이 한계는 리포트에 명시된다. **네이티브 관습 노드 보강은 차기 버전** 과제다.
- **경쟁사 비교 리뷰 미지원(v1)** — 벤치마크 대상 서비스와의 직접 비교 진단은 v1 범위에 없다. 인테이크·워크플로에 비교 섹션을 넣지 않는다. **차기 과제**로 검토한다.
- **디자인 툴 연동은 호스트 의존** — Figma/Pencil MCP·브라우저 조회로 디자인 파일·라이브 화면을 직접 읽는 것은 호스트에 해당 도구가 연결된 경우에만 가능하다. 없으면 스크린샷·텍스트 묘사로 대체하며, 이때 확인 못 하는 항목(실측 대비·동작)은 "검증 필요"로 남는다.
- **맥락 의존** — 서비스·타깃·목표(인테이크)가 부실하면 진단이 일반론에 그친다. `memory/PROJECT.md`의 맥락 구체성이 리포트 품질을 좌우한다.
- 최종 제품·디자인 결정의 책임은 사용자에게 있다.
