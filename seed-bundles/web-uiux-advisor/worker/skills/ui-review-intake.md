# UI 리뷰 인테이크 질문지

리뷰·자문은 화면이 아니라 **맥락**에서 시작한다. 타깃 사용자와 핵심 목표를 모르는 상태의 UI 조언은 취향 품평에 불과하다. 아래 항목이 채워지기 전에는 진단을 시작하지 않는다 — 서비스·타깃·핵심 목표가 비면 **묻고 답을 기다린다.**

**핵심 원칙:** "3초 안에 답할 수 없는 질문은 사용자에게도 어려운 질문이다." 인테이크 질문은 사용자가 바로 답할 수 있게 구체적으로 던진다.

## 질문 구조 (디자인-우선 스펙을 리뷰 인테이크로 각색)

| 카테고리 | 질문 | 왜 필요한가 |
| --- | --- | --- |
| **SERVICE** | 이 서비스는 무엇을 하나? 한 문장으로. | 모든 휴리스틱은 "이 목적에 맞는가"로 판정된다. |
| **TARGET** | 주 사용자는 누구인가? (역할·숙련도·상황) | 익숙한 패턴(Jakob)·정보 밀도·용어 판단의 기준. |
| **GOAL** | 이 화면에서 사용자가 해야 할 **1차 행동**은? | 폴드·CTA·어텐션 비율 판정의 기준(핵심 전환). |
| **PLATFORM** | 웹 / 모바일웹 / 앱? 주 사용 기기는? | 엄지존·터치 타깃·반응형 판정 여부를 가른다. |
| **MARKET** | 대상 시장·언어는? (예: 글로벌 / 한국 / 일본…) | 시장 관습 노드 적용 여부를 가른다(아래 규칙). |
| **TARGET SCREEN** | 리뷰 대상은 무엇인가? 연결된 디자인 툴(Figma/Pencil)·라이브 URL / 스크린샷 / HTML / 텍스트 묘사 / 기획안? | 수집 방법과 시나리오(기획 자문 vs 화면 리뷰) 분기. |
| **CONSTRAINTS** | 지켜야 할 브랜드 가이드·기술 스택·바꿀 수 없는 요소가 있나? | 개선안의 실현 가능성·수정난이도 판정. |
| **SCOPE** | 특정 플로우(가입·결제 등)에 집중할지, 전체 첫인상인지? | 2회 순회의 범위 설정. |

## 진행 규칙

- **필수 3항목**(SERVICE·TARGET·GOAL)이 비면 진단을 시작하지 않고 먼저 묻는다. 나머지는 합리적 기본값을 제안하되 가정임을 표시한다.
- 답이 모호하면("일반 사용자") 한 겹 더 좁힌다("가입 전 방문자인지, 로그인한 재방문자인지").
- 기획 자문 시나리오(화면이 아직 없음)에서는 TARGET SCREEN 대신 기획안·원하는 톤을 묻는다.
- **시장 관습은 조건부 적용** — 기본값은 **시장 중립**이다. MARKET이 특정 시장(예: 한국)으로 확인된 경우에만 해당 시장 관습 노드(예: `korean-market-ux-patterns`)를 적용한다. 확인 전에는 보편 원칙만 쓰고, 한국 관습 등을 임의로 들이대지 않는다.
- **입력 확보 우선순위** — TARGET SCREEN은 **① 연결된 디자인 툴(Figma/Pencil MCP)·라이브 URL 직접 조회 → ② 스크린샷 → ③ 텍스트 묘사** 순으로 확보한다. 상위 수단이 가능한지 먼저 확인하고, 안 되면 다음 수단으로 대체한다.
- 인테이크 결과는 `memory/PROJECT.md`의 현재 자문 맥락에 채울 후보로 정리한다(자동 확정 금지).
