대기는 없앨 수 없어도 **체감 대기**는 설계할 수 있다. 스켈레톤 스크린(콘텐츠 자리를 회색 블록으로 미리 보여줌)은 진행감과 곧 나타날 구조를 예고해 체감 대기를 줄이는 것으로 알려져 있다 — 다만 스켈레톤이 스피너보다 항상 낫다는 근거는 혼재하니 맥락(구조 반복 여부·대기 길이)에 따라 판단한다. **[H10]** · [[doherty-threshold]]

리뷰 관점:

- 데이터 로딩 구간에 아무 피드백이 없으면 "멈춤"으로 오해된다 — 최소한 진행 표시가 있어야 한다.
- 목록·카드·프로필처럼 구조가 반복되는 화면은 스켈레톤이 적합하다. 짧은(수백 ms) 작업엔 오히려 깜빡임을 만들 수 있으니 즉시 낙관적 UI를 쓴다.
- 스켈레톤은 실제보다 빠르게 느껴지게 하지만 실제 성능 문제를 가리는 변명이 되어선 안 된다.
- 정적 화면에서는 로딩 동작을 확인 못 하니 "로딩 상태 검증 필요"로 표기한다.

기다림을 비워 두지 않는 것이 핵심이다.
