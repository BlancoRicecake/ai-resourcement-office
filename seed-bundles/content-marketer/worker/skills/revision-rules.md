# Revision Rules

피드백을 반영할 때 전체를 다시 쓰지 않고 위치를 특정해 해당 구간만 고치는 규칙이다.

## 절차

1. 피드백 원문을 다시 확인한다. 모호하면 무엇을 바꾸려는지 되묻는다.
2. 콘텐츠에서 수정 위치를 특정한다(어느 제목/문단/슬라이드/CTA인지).
3. 피드백 유형을 분류한다.
4. 해당 구간만 수정하고, 무엇을 왜 바꿨는지 한 줄로 남긴다.
5. 수정 구간과 변경 이유를 보여준다. 의미·주장·CTA·발행 조건이 달라질 때만 다시 컨펌받는다.

## 피드백 유형

| 유형 | 다루는 것 | 수정 범위 |
| --- | --- | --- |
| Hook | 후킹, 제목, 첫 줄 | 도입부만 |
| Structure | 목차, 순서, 흐름 | 해당 섹션 배치 |
| Tone | 말투, 브랜드 보이스 | 표현 층위 |
| Evidence | 근거, 수치, 사례 | 해당 주장 + `[검증 필요]` 표기 |
| CTA | 행동 유도 문구·위치 | CTA 부분 |
| Safety | 과장·법적·컨펌 게이트 항목 | 해당 표현, `claim-safety-rules.md` 적용 |

## 금지

- 요청받지 않은 전체 재작성
- 검증되지 않은 수치를 확정 표현으로 바꾸기(`[검증 필요]` 유지)
- 피드백을 근거로 삼아 컨펌 게이트 항목(가격·효능·법적 표현·발행)을 임의 확정하기

## 두 출처의 피드백

- 사용자 피드백은 항상 우선한다.
- 그로스 마케터의 회고 피드백(후킹 방식, 주제, 채널 반응 개선 지시)은 다음 콘텐츠에 반영하고 반영 위치를 기록한다. 사용자 확정 규칙과 충돌하면 사용자를 우선하고 충돌을 알린다.
- 반복해서 확인된 피드백만 `DECISIONS.md 후보`로 제안한다. 자동으로 규칙화하지 않는다.
