# 고객 인터뷰 정리, 팀이 다시 쓰게 만드는 4단계

고객 인터뷰를 마치면 녹취록과 메모는 남습니다. 그런데 다음 제품 회의에서 그 기록을 바로 찾을 수 없다면, 인터뷰에서 얻은 정보는 의사결정과 멀어집니다.

해결의 핵심은 더 많은 내용을 기록하는 것이 아닙니다. 누가 다시 읽더라도 **고객이 무엇을 말했고, 우리는 무엇을 해석했으며, 다음에 무엇을 확인해야 하는지** 알 수 있게 정리하는 것입니다.

이 글에서는 특정 도구보다 먼저 적용할 수 있는 고객 인터뷰 기록의 기본 구조를 소개합니다.

## 인터뷰 기록이 다시 쓰이지 않는 이유

회의가 끝난 직후에는 인터뷰 내용이 생생합니다. 하지만 시간이 지나면 기억에 의존하기 어렵습니다. 이때 긴 녹취록만 남아 있으면 필요한 문장을 다시 찾는 데 시간이 들고, 팀원마다 서로 다른 요약을 만들 수 있습니다.

먼저 현재 기록을 다음 질문으로 점검해 보세요.

- 인터뷰 목적과 핵심 질문을 한눈에 볼 수 있는가?
- 고객의 실제 표현과 팀의 해석이 구분되어 있는가?
- 나중에 주제나 고객 유형으로 검색할 수 있는가?
- 이 인터뷰가 어떤 제품 결정과 연결됐는가?
- 아직 확인하지 못한 가정이 남아 있는가?

여러 항목에 답하기 어렵다면 인터뷰 횟수보다 기록 구조를 먼저 고칠 필요가 있습니다.

## 1단계: 인터뷰 목적과 핵심 질문을 맨 위에 둡니다

기록의 첫 부분에는 인터뷰를 진행한 이유를 짧게 적습니다. 모든 대화를 요약하려 하기보다 이번 인터뷰에서 확인하려던 가정을 분명히 합니다.

예를 들면 다음과 같습니다.

```md
- 인터뷰 목적: 고객 인터뷰 기록이 팀에서 재사용되지 않는 이유 확인
- 대상: 초기 스타트업 PM
- 확인할 질문:
  1. 인터뷰 후 기록을 어디에 보관하는가?
  2. 제품 회의에서 기록을 다시 찾는가?
  3. 다시 찾기 어렵다면 가장 큰 이유는 무엇인가?
```

이 정보가 있으면 나중에 기록을 읽는 팀원도 대화의 맥락을 빠르게 이해할 수 있습니다.

## 2단계: 고객의 말과 팀의 해석을 분리합니다

고객이 말한 사실과 팀이 내린 해석을 같은 문장에 섞으면, 시간이 지난 뒤 무엇이 실제 발언이었는지 구분하기 어렵습니다. 기록을 세 칸으로 나누면 이 문제를 줄일 수 있습니다.

| 구분 | 기록할 내용 | 예시 |
| --- | --- | --- |
| 고객 발언 | 고객이 실제로 말한 문장 | “지난 인터뷰를 찾으려면 문서를 하나씩 열어봐야 해요.” |
| 관찰·맥락 | 발언이 나온 상황 | 제품 회의 전에 이전 인터뷰를 찾는 과정을 설명함 |
| 팀의 해석 | 검증이 필요한 가설 | 검색보다 문서 위치를 기억하는 데 의존하고 있을 수 있음 |

고객의 직접 인용을 외부 콘텐츠에 사용할 때는 공개 동의 여부를 확인해야 합니다. 동의가 없다면 개인과 회사를 특정할 수 없게 익명화하고, 의미가 달라지지 않도록 주의합니다.

## 3단계: 다시 찾을 수 있는 태그를 최소한으로 붙입니다

태그가 너무 많으면 기록하는 사람이 지치고, 너무 적으면 검색이 어렵습니다. 처음에는 팀이 실제로 다시 찾을 기준만 사용합니다.

- 고객 유형: 창업자, PM, 마케터 등
- 문제 영역: 인터뷰 정리, 팀 공유, 제품 결정 등
- 여정 단계: 탐색, 도입, 사용, 이탈 등
- 확인 상태: 한 번 관찰, 반복 관찰, 추가 확인 필요

태그 이름은 팀 안에서 같은 의미로 사용해야 합니다. `공유 문제`, `협업 문제`, `팀 전달 문제`처럼 비슷한 태그가 늘어나면 하나로 합칩니다.

## 4단계: 기록 끝에 결정과 다음 행동을 남깁니다

좋은 인터뷰 기록은 요약에서 끝나지 않습니다. 이 인터뷰로 무엇을 결정했고 무엇을 더 확인할지 남겨야 다음 회의에서 사용할 수 있습니다.

```md
## 결정 연결

- 이번 인터뷰만으로 확정한 것: 없음
- 추가로 확인할 가정: 검색 기능이 부족해서 기록을 다시 보지 않는가?
- 다음 인터뷰 질문: 최근 제품 회의에서 과거 인터뷰를 찾은 과정을 보여달라고 요청
- 연결된 제품 결정: 인터뷰 검색 화면의 탐색 방식 검토
- 담당자와 확인 시점: PM / 다음 스프린트 계획 전
```

한 명의 발언을 전체 고객의 요구로 일반화하지 않는 것도 중요합니다. 같은 문제가 반복되는지 확인하고, 반대 사례가 있는지도 함께 기록합니다.

## 팀이 다시 쓰게 만드는 회의 습관

기록 형식을 바꿔도 업무 흐름에서 다시 꺼내지 않으면 활용하기 어렵습니다. 제품 회의 문서에 다음 항목을 추가해 보세요.

- 이번 결정과 연결된 고객 인터뷰
- 결정을 지지하는 발언과 반대하는 발언
- 아직 근거가 부족한 가정
- 결정 후 다시 확인할 지표 또는 인터뷰 질문

이 방식은 인터뷰를 “참고 자료”로만 두지 않고, 어떤 근거로 결정했는지 추적하는 데 도움을 줄 수 있습니다.

아래 구조로 작게 시작할 수 있습니다.

```md

- 대상:

- 확정한 것:

- 다음 질문:
- 연결된 제품 결정:
- 담당자 / 시점:
```

## 자주 묻는 질문

### 인터뷰를 몇 개부터 체계적으로 정리해야 하나요?

정해진 개수보다 팀이 기록을 다시 찾기 어려워지는 시점이 기준입니다. 처음부터 복잡한 분류 체계를 만들기보다, 다음 회의에서 실제로 찾을 기준부터 정하는 편이 관리하기 쉽습니다.

### AI로 인터뷰를 요약해도 되나요?

초안 요약과 주제 분류에는 활용할 수 있습니다. 다만 고객의 직접 인용, 민감정보, 제품 결정에 쓰는 핵심 해석은 사람이 원문과 대조해야 합니다. 외부 모델에 자료를 보낼 때는 개인정보와 회사의 데이터 처리 기준도 확인하세요.

### 팀마다 요약 방식이 다른 문제는 어떻게 줄이나요?

공통 템플릿과 최소 태그를 먼저 합의하세요. 이후 실제 회의에서 쓰이지 않는 항목은 제거하고, 반복해서 필요한 항목만 남기는 방식으로 조정할 수 있습니다.

## 정리

고객 인터뷰를 다시 쓰기 위해 필요한 것은 긴 요약문이 아니라 일관된 연결 구조입니다.

1. 인터뷰 목적과 핵심 질문을 적습니다.
2. 고객의 말과 팀의 해석을 분리합니다.
3. 다시 찾을 수 있는 최소 태그를 붙입니다.
4. 제품 결정과 다음 확인 행동을 연결합니다.

- [ ] CMS 미리보기에서 제목 구조·표·코드 블록·모바일 줄바꿈 확인
- [ ] 링크드인 문구의 URL과 CTA 확인
- [ ] 주장 검증표의 `검증 필요` 항목 모두 처리
- [ ] 사용자 최종 승인 후 게시

## 7. 사용자 컨펌 지점

- 통합 제작안과 권장 제목
- 고객 문제 진술의 공개 가능 범위
- 제품 기능·무료 체험 CTA 문구
- 블로그와 링크드인 최종 게시 여부

## 8. 성과 기록 안내

발행 후 2주 또는 실험 종료 시점에 다음 데이터를 제공하면 결과 보고서를 만들 수 있다.

- 블로그 방문수와 CTA 클릭수
- 무료 체험 신청 수
- 링크드인 노출·클릭·반응
- 채널별 URL과 발행일

데이터가 없으면 성과를 추정하지 않고 `판단 불가`로 기록한다.

## 9. 메모리 업데이트 후보

```md
### EDITORIAL.md 후보
- [추가] 필러: 고객 인터뷰 운영 — 인터뷰를 실행과 제품 결정에 연결하는 방법
- 이유: 이번 콘텐츠의 상위 주제이며 후속 주제로 확장 가능하다.

### PROJECT.md 후보
- [추가] 타깃 독자: 초기 스타트업 PM, 창업자 / 톤: 전문가형+친근
- 이유: 작업 지시서에서 명시적으로 제공됐다.

### RESEARCH.md 후보
- [추가] 고객 인터뷰 기록 재사용 문제는 사용자 제공 가설이며 외부 공개 전 고객 자료 확인 필요
- 이유: 확인되지 않은 고객 문제를 일반 사실처럼 반복하지 않기 위해서다.

### CONTENT-LOG.md 후보
- [추가] 발행 후 블로그와 링크드인 게시물의 URL·성과 기록
- 이유: 연결 실험의 결과 판단과 다음 콘텐츠 개선에 필요하다.
```

사용자가 승인한 항목만 각 메모리 파일에 반영한다.
