# Memory Loading Rules — 메모리 적재 규칙

메모리는 모델 학습이 아니라, 다음 세션에서 다시 읽어 적용하는 운영 기록이다. 훈련소는 신임 에이전트의 메모리에 배경지식을 채우되, 확정은 사용자 승인 후에만 한다.

## 적재 대상 파일 (신임 에이전트 쪽)

| 파일 | 들어가는 내용 |
| --- | --- |
| `memory/CURRICULUM.md` | Phase 2에서 번역한 실무 지침(직무 배경지식). 훈련소의 최종 산출물. |
| `memory/knowledge/<node-id>.md` | 세션·직무를 넘어 재사용되는 **일반화된 지식 노드** (아래 노드 적재 규칙) |
| `memory/knowledge/INDEX.md` | 노드 색인 — 노드를 적재하면 반드시 한 줄 동반 갱신 |
| `memory/RESEARCH.md` | 출처·근거(참고문헌만, 원문 없음), 검증 후보 지식 |
| `memory/DECISIONS.md` | 훈련 라운드에서 채택/보류/제외한 지식과 이유 |

신임 에이전트에 해당 파일이 없으면 생성 후보로 제안한다. 도메인 지식 파일 이름이 이미 있으면 그 이름을 따른다.

## 지식 노드 적재 규칙 (bundle-standard §2.5 컨벤션)

Phase 2 번역 결과 중 **일반화 가능한 지식**(특정 훈련 대상·프로젝트를 넘어 재사용되는 원리·프레임워크)은 CURRICULUM.md 항목으로만 남기지 않고 노드로도 적재한다.

- 노드 형식: frontmatter(`id`(=파일명)·`title`·`tags`·`source: seed`·`confidence`·`created`) + 본문 + 관련 노드로의 `[[위키링크]]`. 훈련소가 배포 전에 적재하는 지식은 검증을 거치므로 `source: seed`, confidence는 근거 강도에 따라 정한다(근거 약하면 `low` + RESEARCH.md 검증 후보로).
- `INDEX.md`에 한 줄(`- [[id]] 제목 — 태그: …`)을 동반 갱신하고, 원 CURRICULUM.md 항목에는 `→ [[node-id]]` 역참조를 남긴다.
- 같은 개념의 노드가 이미 있으면 새 노드를 만들지 않는다 — 기존 노드의 보강/`confidence` 상향 후보로 제시한다.
- 노드 본문에도 원문 발췌를 넣지 않는다. 출처는 짧은 표기만(상세는 RESEARCH.md).

## 적재 원칙

- 원문 텍스트는 버린다. Phase 2에서 새로 쓴 실무 지침만 적재한다.
- 메모리는 자동 확정하지 않는다. 훈련소는 후보를 제안하고, 사용자가 승인한 것만 파일에 반영한다.
- 기존 메모리를 덮어쓰지 않는다. 충돌 시 병합/교체 후보를 나란히 제시한다.
- 오래되거나 틀린 지식은 교체·삭제 후보로 표시한다.
- 메모리 파일에 API 키, 비밀번호, 개인정보, 원문 발췌를 넣지 않는다.

## 적재 후보 표기 형식

```txt
[적재 후보] 대상 파일: memory/CURRICULUM.md
- 추가할 내용: (요약)
- 기존 내용과 충돌: 없음 / 있음(내용)
- 처리 제안: 추가 / 병합 / 교체
- 근거 출처: (RESEARCH.md에 함께 적재)
```

지식 노드일 때:

```txt
[적재 후보] 대상 파일: memory/knowledge/<node-id>.md (신규 / 기존 보강)
- 내용: (한 줄 요약 + 주요 [[링크]] 대상)
- 기존 노드와 충돌: 없음 / 있음(내용)
- 처리 제안: 추가 / 병합(confidence 상향) / 교체
- INDEX.md 동반 갱신: 1줄 추가 (또는 갱신)
- CURRICULUM.md 역참조: → [[node-id]]
- 근거 출처: (RESEARCH.md에 함께 적재)
```

## 훈련 완료 기준

- 커리큘럼의 각 주제에 실무 지침이 1개 이상 적재됐는가?
- 일반화 가능한 지식이 노드로 적재되고 INDEX.md·CURRICULUM 역참조가 함께 갱신됐는가?
- 각 실무 지침에 재확인용 출처가 연결됐는가?
- 원문·긴 인용문이 메모리에 남지 않았는가?
- 고위험 직무면 전문가 검토·규제 확인 문구가 함께 적재됐는가?
- 사용자가 적재를 승인했는가?
