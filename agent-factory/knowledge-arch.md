# 지식 아키텍처 — 4층 컷 · 지식 노드 그래프 · 자가학습

> **결정(2026-07-10): 그래프는 "엔진 없이 데이터만"이 표준이다.** 지식 그래프의 두
> 부분을 분리한다 — **데이터**(= `[[위키링크]]` 달린 마크다운 노드 + `INDEX.md`)는
> 모든 번들의 표준 컨벤션(`bundle-standard.md` §2.5)이고, **엔진**(검색·탐색을 계산하는
> 코드, `search_knowledge` 등)은 코드형 번들만 얹는 선택 자동화(부록 A 개념)다.
> 엔진이 없어도 그래프는 동작한다: 에이전트가 INDEX를 읽고 필요한 노드만 연 뒤
> `[[링크]]`를 따라간다. 이것이 "전부-로드 비용이 누적량에 비례"하는 문제를 런타임 0으로
> 푸는 방법이다 — 비용은 누적량이 아니라 "지금 필요한 양"에 비례하게 된다.

에이전트가 "정해진 순서로 툴만 호출하는 스크립트"가 아니라 **그 분야 전문가처럼
판단하고, 쓸수록 성장하는** 존재가 되게 하는 지식 설계다. Hermes(컨텍스트 파일
주입식 학습)에서 출발하되, 정적 지식을 **검증된 그래프**로, 동적 학습을 **업데이트
불변 뿌리**로 분리해 더 낫게 만들었다.

## 1. 4층 컷 (무엇을 어디에 두나)

| 층 | 내용 | 위치 | 로딩 |
|---|---|---|---|
| L1 | 정체성 + 상위 휴리스틱 | `worker/agent.md` | 항상 |
| L2 | 계산 규칙(결정론) | `app/core` | 코드 실행 |
| L3 | 도메인 지식 그래프 | seed + learned 2뿌리 | 온디맨드 조회 |
| L4 | 절차(경험) | `worker/skills/` + `save_workflow` | 온디맨드 |

원칙: **자주·항상 필요한 것일수록 위층(항상 로드), 방대·간헐적일수록 아래층
(조회)**. agent.md는 짧게 유지하고 세부 지식은 그래프로 내린다.

## 2. 2-뿌리 지식 그래프 (핵심)

지식 = 마크다운 파일이 **노드**, `[[wikilink]]`가 **엣지**. 하나의 조회 뷰,
물리적으로 두 뿌리:

| 뿌리 | 위치 | 성격 | 번들 업데이트 시 |
|---|---|---|---|
| **seed** | `worker/knowledge/*.md` (번들 내) | 큐레이션·**검증됨**·읽기전용 | **교체됨** |
| **learned** | `~/.<slug>/knowledge/*.md` (성장 레이어) | 런타임 자가학습·출처태깅 | **살아남음** |

`search_knowledge`/`knowledge_neighbors`는 **두 뿌리를 병합한 그래프**를 조회한다.
→ 에이전트가 종료됐다 재실행돼도 learned 노드가 그대로 검색된다(**연속성이 핵심**).

**왜 두 뿌리를 분리하나:**
1. `worker/knowledge/`는 번들 업데이트 시 덮인다 → 학습분을 거기 쓰면 소실. learned는
   업데이트 불변 홈에 둔다.
2. 출처 보존 — seed(검증) vs learned(자가생성)를 구분해 품질 게이트를 지킨다.
3. 배송 아티팩트를 재현·검증 가능하게 유지.

**노드 형식(두 뿌리 동일):**
```markdown
---
id: hero-five-elements        # kebab-case, 파일명과 동일, 유일
title: 히어로 5요소
tags: [above-fold, structure]
source: seed                  # seed | learned
confidence: high              # seed=high; learned=low에서 시작
created: 2026-07-09           # learned 전용
evidence: []                  # learned: 근거 문자열/URL
---
본문. 다른 노드는 [[attention-ratio]] 로 연결 = 엣지.
```

**빌드 검증 게이트(필수, 테스트로 강제):**
- seed의 모든 `[[wikilink]]`가 존재하는 seed id로 해소(dangling 0)
- self-loop 없음, 중복 id 없음, id == 파일명
- `graph.json` 재빌드 == 커밋본(동기화)

## 3. 자가학습 메커니즘

정적 지식(설계 시 큐레이션)과 동적 성장(런타임 축적)은 **다르다**. 성장이 실제로
일어나려면 도구뿐 아니라 **행동 강령**이 있어야 한다.

**기계(machinery):**
- `remember` — 사용자·브랜드 종속 사실(페르소나 선호, 브랜드 보이스, 금지표현, 결정)
- `save_workflow` — 채택된 재사용 절차(L4 learned)
- `learn_knowledge` — 범용 도메인 지식 → learned 뿌리(L3 learned)
- 저장 위치 = `~/.<slug>/` (업데이트 불변), **사용자 우선 병합**, **위생 필터**
  (PII·미공개 매출/전환 거부, 우회 금지)

**행동 강령(agent.md §자가학습에 반드시 주입):**
1. **조회 우선** — 작업 시작 전 관련 주제를 `search_knowledge`로 조회, 필요 시
   `knowledge_neighbors`로 확장해 판단 근거로 삼는다.
2. **학습 기록** — 일반화 가능한 도메인 사실/패턴을 새로 확인하면 `learn_knowledge`로
   learned 뿌리에 기록하고, 무엇을 기록했는지 사용자에게 표시한다.
3. **자율 스킬 생성·개선(B)** — 복잡한 작업을 **성공적으로 마치면 사용자 요청이
   없어도** 재사용 가능한 절차를 `save_workflow`로 증류한다. 기존 스킬과 겹치면
   새로 만들지 말고 **그 스킬을 개선·갱신**한다(스킬 자기개선). Hermes의 "완료 시
   자동 스킬 생성 + 사용 중 개선"에 대응한다.
4. **구분** — `remember`(이 사용자·이 브랜드) vs `learn_knowledge`(범용 도메인) vs
   `save_workflow`(재사용 절차).
5. **learned는 성장 레이어에만** — 번들 `worker/knowledge/`에 쓰지 않는다(업데이트 시 덮임).
6. **위생 동일 적용** — knowledge에도 PII·미공개 매출 기록 금지.

### 3.5 정리와 승인 (consolidation & approval, C)

닫힌 루프가 오염·비대해지지 않게 하는 장치. Hermes의 "agent-curated memory + 주기적
nudge", "write_approval"에 대응한다.

- **정리(consolidation)** — 같은 사실/절차를 반복 확인하면 새 노드를 만들지 말고
  기존 노드의 `confidence`를 올린다(예: low→med→high). 근접 중복은 병합하고
  (`memory-log`의 in-place 병합 패턴을 그래프로 확장), 서로 모순되는 learned 노드는
  `conflict` 플래그를 달아 사용자에게 확인한다. 세션 경계에서 "새로 배운 것 정리할까요?"
  같은 **가벼운 주기적 nudge**로 에이전트가 스스로 큐레이션하게 한다.
- **승인(write_approval, 선택)** — 기본은 지금처럼 즉시 기록 + "무엇을 기억했는지 표시"
  + 위생 자동 거부다. 여기에 **선택적 승인 스테이징**을 둔다: 민감하거나 광범위한
  learned 쓰기는 다음 세션에 영향을 주기 **전에** 사용자 확인을 받도록 대기시킬 수
  있다(`confidence: pending`). 사용자가 확인하면 활성화, 거부하면 폐기.
- **에피소드 아카이브(A)는 현재 보류** — Hermes의 3번째 메모리 층(SQLite FTS5 전체
  세션 이력 + LLM 요약 크로스세션 리콜)은 아직 표준에 넣지 않는다. 대체로 append-only
  `decisions.log` + `search_knowledge`를 쓴다. 도메인이 원시 이력 검색을 요구하면
  선택 스펙으로 추가한다(향후 과제).

## 4. 승격(promotion) — 팩토리로의 피드백 (선택, 런타임 아님)

반복 확인된 learned 지식은 **팩토리에서** 검증 게이트(인용 해소·엣지 정합)를 통과시켜
다음 번들 버전의 seed로 승격할 수 있다. 런타임에 사용자 머신에서 자동 승격하지
않는다 — 배송 지식의 검증성을 지키기 위해서다.
